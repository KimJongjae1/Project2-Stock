## 🧠 상세 내용 정리

### 1. 대표적인 아키텍처 구조

1) MSA(마이크로서비스 아키텍처)
- 정의 : 애플리케이션을 여러 개의 독립적인 서비스로 분리한 구조
- 특징 : 큰 프로젝트에 적합
- 장점
    (1) 각 서비스의 독립적 개발/배포/확장 가능
    (2) 하나의 서비스에서 장애가 발생해도 다른 서비스에 영향이 적음
    (3) 조직을 서비스 단위로 분리해서 효율적인 협업 가능
- 단점
    (1) 초기 설계와 운영 비용이 높을 수 있음
    (2) 서비스 간 통신 등으로 인해 시스템이 복잡
    (3) 트랜잭션 관리와 데이터 일관성 확보가 어려움

2) MA(모놀리식 아키텍처)
- 정의 : 모든 기능이 하나의 코드베이스와 애플리케이션으로 통합된 구조
- 특징 : 작은 프로젝트에 적합
- 장점
    (1) 초기 개발 속도가 빠름
    (2) 개발 및 배포가 단순함
    (3) 단일 환경에서 실행되므로 복잡한 인프라 필요 X
- 단점
    (1) 규모가 커질수록 유지보수와 확장이 어려움
    (2) 작은 변경에도 전체 애플리케이션을 다시 배포해야 함
    (3) 특정 서비스의 장애가 전체 서비스에 영향을 줄 수 있음

### 2. 대표적인 패키지 구조

1) 계층형 구조(Layered Architecture)
- 정의 : 소프트웨어를 역할별로 계층을 나누는 구조
- 특징 : 계층 간 독립성이 높음
- 장점
    (1) 각 계층이 명확한 책임을 가져서 유지보수와 테스트가 쉬움
    (2) 전통적으로 많이 써왔던 구조여서 익숙한 개발자가 많음
    (3) 프로젝트에 대한 이해도가 낮아도 전체적인 구조 파악이 쉬움
- 단점
    (1) 도메인의 응집도가 낮음
    (2) 계층 간 의존성이 많아지면 유연성이 떨어질 수 있음
    (3) 규모가 커지면 하나의 패키지 안에 여러 클래스가 모여서 구분이 어려워짐

2) 도메인형 구조(Domain-Oriented Architecture)
- 정의 : 비즈니스 도메인 중심으로 코드를 분리하는 구조
- 특징 : 기능별로 독립적인 배포/테스트가 가능
- 장점
    (1) 도메인 단위로 개발과 변경이 쉬움
    (2) 도메인 지식이 코드 구조에 잘 녹아듦
    (3) 도메인 별로 나눠져 있어서 팀 단위 개발에 유리함
- 단점
    (1) 도메인 구분이 명확하지 않을 경우 구조가 복잡해질 수 있음
    (2) 공통된 코드(유틸리티, 공통 서비스 등)가 분산될 수 있음
    (3) 프로젝트에 대한 이해도가 낮으면 전체적인 구조 파악이 어려움
